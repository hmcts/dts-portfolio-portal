# ADR-003: AI parser

- **Status:** Accepted
- **Date:** 2026-05-19
- **Deciders:** DTS Portfolio Portal team

## Context

The portal accepts loosely-structured markdown uploads and uses an AI parser to translate them into structured entity data (§7.5). The same AI service is used at search time to synthesise the one-sentence answer card (§5.7, §6.1). The parser must support structured output (JSON schema or function calling), per-field confidence flags, idempotent parsing for identical input, and a clean failure mode when the service is unavailable (§7.5 — the strict-template parser fallback).

The stack-design spec (§4, §10) locks Azure OpenAI as the AI provider. This ADR records that decision.

## Options considered

### Option A — Azure OpenAI

Microsoft-hosted, in-tenancy LLM service. Structured-output mode supported (JSON schema and function calling). Data residency aligns with HMCTS gov-cloud requirements. Accessed via managed identity (`DefaultAzureCredential`) — no API keys to leak. Selected.

### Option B — Anthropic Claude direct

Strong instruction-following and structured-output support. Rejected: data residency requires a separate procurement and contractual review path that Azure OpenAI does not (already in the HMCTS tenancy).

### Option C — OpenAI direct

Same structured-output capability as Azure OpenAI; same data-residency concern as Option B. Rejected.

### Option D — Self-hosted (Llama / similar)

Maximum data control, no per-parse cost. Rejected: infrastructure burden incompatible with v1 timeline; weaker structured-output compared with Azure OpenAI's function-calling pathway.

## Decision

Azure OpenAI. Access via App Service user-assigned managed identity using `DefaultAzureCredential`. Structured-output mode (JSON schema matching entity types defined in Phase 1) for the parse step. Caching by source-markdown hash to avoid re-billing for identical re-uploads (§7.5 idempotency requirement).

## Consequences

- Per-parse cost — ops needs an Azure OpenAI quota allocation and budget alert.
- Strict-template parser fallback (per §7.5) is a Phase 2 task — handles the AI-unavailable case.
- Cache parses by source-markdown SHA-256 in the audit log row to enable idempotent re-processing.
- No API keys in code or config; identity model is the same as for Postgres and Key Vault.
- Answer-card synthesis (Phase 3) uses the same client and the same identity — one service to monitor.
